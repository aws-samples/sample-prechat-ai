## 1. UI 커스텀 시 화면 렌더링 시작점 개선

### 현황

- 아직 커스텀 UI 환경 정보가 전달되지 않았을 경우, 프론트엔드는 **기본 UI**를 먼저 노출한다.
- 이후 커스텀 UI 환경 정보가 도착하면 화면이 전환되며 깜빡임(Flash of Default Content)이 발생한다.

### 개선 방향

- 커스텀 UI 환경 정보 수신 전까지 렌더링을 지연하거나, 로딩/스켈레톤 상태로 처리할 수 있는지 검토.
- **커스텀 UI가 없는 케이스**(캠페인에 커스텀 설정이 없는 경우)에도 정상 대응할 것.
  - 즉, "환경 정보 로드 완료 여부"와 "커스텀 UI 존재 여부"를 분리된 상태로 관리해야 함.

### 참고 파일(추정)

- `packages/web-app/src/contexts/` 내 Customization Context
- `packages/backend/admin/customization_handler.py`

## 2. 캠페인 메트릭 개선 — 상담 목적 파이차트

### 현황

- 상담 목적(Consultation Purpose) 파이차트에서 `|` 구분자로 다중 목적이 연결된 아이템이 **하나의 단일 카테고리**로 집계됨.
- 예: `"기술 문의 | 가격 문의"`가 단일 슬라이스로 표시됨.

### 개선 방향

- `|` 구분자로 분절(split)하여 각 목적을 **독립 카운트**로 집계.
- 하나의 세션이 N개의 목적을 가진 경우, 각 목적에 +1씩 카운트 (중복 허용).
- 집계 결과는 파이차트/도넛차트 등 시각화에 반영.

### 참고 파일(추정)

- `packages/backend/campaign/campaign_analytics.py`
- 프론트엔드 캠페인 분석 화면 (관리자 대시보드)

## 3. 인바운드 캠페인 신설

### 3.1 도메인 용어 변경

현재 구현된 캠페인을 **아웃바운드 캠페인(Outbound Campaign)**으로 재명명하고,
새로운 유형인 **인바운드 캠페인(Inbound Campaign)**을 도입한다.

| 구분 | 아웃바운드 캠페인 (기존) | 인바운드 캠페인 (신규) |
|------|-------------------------|----------------------|
| 세션 생성 주체 | 영업 담당자 (Sales Rep / CSR) | 고객 스스로 (Self-service) |
| 고객 접근 방식 | 담당자가 세션 URL 전달 | 캠페인 URL + PIN 직접 접근 |
| PII 수집 | 담당자가 세션 생성 시 입력 | 고객이 직접 입력 |
| 세션 할당 | 1:1 사전 배정 | 동적 자가 할당 |

### 3.2 인바운드 캠페인 동작 흐름

1. 관리자가 인바운드 캠페인 생성 → **캠페인 URL + PIN 코드** 발급
2. 익명 유저가 `인바운드 캠페인 URL + PIN 코드`에 접근
3. 유저가 본인 PII 정보(이름, 이메일, 회사, **전화번호**)를 직접 입력
4. 시스템이 해당 유저를 위한 세션을 자동 생성
5. 유저가 에이전트와 대화 시작

### 3.3 고유성(Uniqueness) 규칙

- 익명 유저의 보안 신원(Principal)은 **별도 구현하지 않음**
  - Cognito Identity Pool, OIDC 연동 **미구현**
  - SMS 인증 **미구현**
- 캠페인 내 프린시팔 고유성 확보를 위해 **전화번호를 수집**
- 동일 전화번호는 **캠페인당 최대 1개의 세션**만 생성 가능 (중복 세션 생성 차단)

### 3.4 프론트엔드 변경사항

1. **메뉴 분할**: "인바운드 캠페인", "아웃바운드 캠페인" 메뉴 분리
2. **인바운드 캠페인 관리 UI**: 생성, 관리, 추적 기능 구현
3. **Agent 연결 설정**: 인바운드 캠페인 단위로 Agent 연결 구성
4. **접근 자산 생성**: 캠페인 URL, PIN 번호, **QR 코드** 생성 기능
5. **익명 유저 세션 생성 플로우**:
   - 캠페인 접근 → PIN 입력 → PII 입력(전화번호 포함) → 세션 생성 → 채팅
6. **관리자 세션 목록**: 익명 유저가 생성한 세션도 동일하게 조회 가능

### 3.5 백엔드 변경사항

1. **도메인 용어 변경 전파**:
   - `Campaign` → `OutboundCampaign`으로 리네임
   - 코드, API, 스키마 전반에 반영
2. **인바운드 캠페인 도메인 신설**:
   - `InboundCampaign` 엔티티 및 핸들러 추가
   - 전화번호 기반 세션 중복 방지 로직
3. **DynamoDB 테이블 신설 여부**: **검토 필요**
   - 기존 `CampaignsTable` 확장 vs. 신규 테이블 분리 비교 분석 필요
   - 백엔드 로직은 **최대한 재사용** 방침
   - 핵심 차이는 프론트엔드 유입 흐름이므로, 백엔드는 경량 보강 수준으로 예상

### 3.6 아키텍처 예상 구도

```
인바운드 유입 흐름(신규)
  익명 유저 → 캠페인 URL + PIN → PII 입력 → 세션 자동 생성 → 대화
                                    ↓
                        전화번호 중복 체크 (캠페인 스코프)

아웃바운드 유입 흐름(기존, 용어만 변경)
  영업 담당자 → 세션 사전 생성 → URL 전달 → 고객 접속 → PIN 인증 → 대화
```
